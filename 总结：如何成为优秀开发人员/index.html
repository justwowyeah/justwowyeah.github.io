总结：如何成为优秀开发人员

发表于 2023-09-24 分类于 学习 ， 总结 本文字数： 3.6k 阅读时长 ≈ 13 分钟

本文根据“如何成为优秀开发人员？”ーー编程随想系列博文总结而成，建议阅读原文
本文提供给回顾原文大纲和快速查看建议的人群

何为优秀？

  • 团队中的少数人，二八原理中的20%
  • 让team leader事事省心
    1. 更快的开发效率
    2. 更好的程序设计
    3. 更好的代码质量
    4. 更善于debug
    5. 更能够解决技术难题
  • 贡献总和与另外80%的贡献总和不相上下，甚至可能超过

优秀开发人员不等于开发大牛

关于兴趣

兴趣是最好的老师

有兴趣的开发人员：

  • 有动力学习新东西
  • 碰到新鲜玩意喜欢刨根问底
  • 碰到有开发过程的困难（比如一些难调试的bug）显得比较有耐心

如果你属于以下情况之一：

  • 即将进入学校学习软件专业
  • 已经毕业，即将入行的新手菜鸟
  • 已经工作若干年，但还不属于优秀开发人员
  • 已经在其它行业工作若干年，想转入软件行业

并且企图在将来成为一个优秀开发人员，那么首先要判断自己是否确实喜欢软件开发

判断是否喜欢软件开发

假设有两个工作岗位A和B供你选择
岗位A：可以随意去干除软件开发外的任何事情
工岗位B：必须全职从事软件开发，不能干其它事情
岗位A的收入比岗位B高很多

如果选择B，那么你确实对软件开发非常热衷

如果对软件开发兴趣不大，有什么方法能让我提高兴趣？

关于提高兴趣

  • 大部分人的基本素养形成于20岁之前。之后不会有太大改变。如果你已从学校毕业，又工作了若干年，那么兴趣多半定型，改变的机会和效果不大（但也不是绝对不可能改变）
  • 兴趣是自然形成的，主观改变自己或他人的兴趣，效果并不理想。与其这样，不如找一个真正感兴趣的行业
  • 如果你年龄尚小（不到20岁），还在读中学（甚至小学），那现在不必考虑“如何成为优秀开发人员”，第一重要的是发现自己的兴趣，并充分发挥它

关于自学能力

自学的重要性

  • IT行业知识更新速度快，很多技术在读书时还没有发明
  • 即使已经发明，老师也未必会教（或许他/她也不懂）
  • 即使老师教，将来工作需要的新技术未必是老师教的那个

自学的主动性

按照自学的主动性不同，国内开发人员可分为如下几类：

  1. 抗拒自学者
    • 不愿意自学，出于懒惰或抵触新事物
    • 需要自学时找理由推诿
    • 占比例不多
    • 如果你属于这类人，建议放弃
  2. 被动自学者
    • 平时不会自学新技术。被逼无奈才勉为其难学习
    • 如果你属于这类人，建议放弃
  3. 需求驱动型自学者
    • 自学的动机和方向基于需求驱动
    • 需要使用新技术就利用业余时间查找资料
    • 如果你属于这类人，考虑向第4类人转型
  4. 计划型自学者
    • 自学的动机和方向基于规划
    • 定期寻找知识结构的缺陷、考虑将来的发展方向、思考新技术的未来，然后制定学习计划
    • 如果你属于这类人，恭喜你

自学的常用招数

搜索引擎

  • 使用搜索引擎是互联网时代的必备基本功

百科类网站

  • 特性：专业术语一般可以查到比较具体的解释
  • 局限性：需要深入了解技术时远远不够

订阅“BBS、Mailing List、Blog”

  • 介绍：通过订阅专业的、针对某个领域的“BBS、Mailing List、Blog”，可以了解该领域的实时动态、热点话题、发展方向。如果碰到疑难杂症还可以提问（运气好还能交几个朋友）
  • 特性：使用订阅可以让信息自动跑到你面前，省去了打开浏览器访问网站的麻烦（因此也节省了时间）。具有一定的交互性
  • 局限性：难以【系统性】掌握比较复杂的技术（比如学习有一定复杂度的编程语言）

看书（电子书/纸版书）

  • 系统性地掌握复杂技术的首选方法是找一本针对性的好书
  • 每一个具体的领域都有N本书可供选择，如何取舍非常重要
  • 利用搜索引擎或者专门的网站（例如豆瓣、亚马逊）来识别好书与坏书
  • 电子书优于纸质书：
    1. 电子书资源丰富，大部分国外出版的IT书籍可以在Internet找到免费电子版
    2. 电子书便于携带、能全文搜索、能共享、能备份、省钱
  • 英文版优于中文版：
    1. 国内懂开发又文笔好的IT作家寥寥无几
    2. 中文书籍要么翻译国外（一般有1-2年的滞后，翻译质量也未必好），要么粗制滥造

设定个人发展目标

什么是个人发展目标

与个人的职业发展有关的目标，包括知识、技能、工作岗位等

如何设立目标

  • 难易适中
    太容易帮助不大，太难容易放弃或超出时间导致打乱计划
  • 容易评估
    否则不清楚目标算不算已达到

个人发展目标的三种类型

短期目标

  • 时间跨度：几个星期到一个季度
  • 需要具体，便于评估

eg：

  • 在本月读完《Thinking in C++》
  • 在本月熟悉 Spring 框架
  • 在这2个月用 C++ 写一个五子棋游戏

中期目标

  • 时间跨度：几个季度到1-2年
  • 比短期目标抽象，必须是短期目标的有机结合

eg：

  • 短期目标是“本周看完《Dive into Python》”，那么中期目标可以是“1年内成为熟练的 Python 程序员”

长期目标

  • 时间跨度：5年以上
  • 与具体的语言、平台等无关，一般和职业岗位关联

eg：

  • 5-7年内成为技术总监
  • 5年内成为公司产品的架构师

个人发展计划

3种目标都设定好之后，就形成了【个人发展计划】：

  • 每一个阶段结束时进行总结，评估目标的完成情况、收获、经验教训
  • 对后续目标进行调整
  • 定期回顾能够获得满足感，有利于坚持计划

优秀的开发人员往往具有较高的效率。【效率】包括两方面：

  • 做正确的事
  • 正确地做事

“做正确的事”比“正确地做事”更加重要

做正确的事

一些不好的习惯

大部分人具有如下【不好】的工作习惯：

  • 先做自己喜欢的事情，再做自己不喜欢的事情
  • 先做紧急的事情，再做不紧急的事情
  • 先做容易做的事情，再做不容易做的事情
  • 先做自己了解、熟悉的事情，再做自己不了解、不熟悉的事情
  • 先做有趣的事情，再做枯燥的事情
  • 先做易于告一段落的事情，再做不易于告一段落的事情
  • 先做自己熟悉的人托付的事情，再做自己不熟悉的人托付的事情

如何克服

评估权重

  • 评估前先设定目标，上文已经谈到
  • 权重来源于这件事情对于达成目标是否有帮助，帮助有多大
    帮助越大则权重越大，帮助越小则权重越小

严格按照权重执行

  • 每天早晨把当天准备做的事情根据权重排好优先级，然后【严格】按照优先级顺序执行
  • 偶尔碰上突发情况也【不要】轻易改变计划，而是先冷静评估事情的权重，只有紧急且权重高的突发事件才能被临时加入计划

正确地做事

“正确地做事”讨论的是【效率】和【质量】的关系，也就是如何“多、快、好、省”地完成工作

善用工具

编辑器

Emacs、VIM、Eclipse等满足以下所有要求

字体

  • 等宽，如 FixedSys、Courier New
  • 能够清楚区分如下字符：
    数字0　和　字母O
    数字1　和　大写字母I　和　小写字母l　和　或运算符|
    数字2　和　字母z
    数字9　和　字母q
    乘号*　和　字母X
    分号;　和　冒号:

颜色

  • 支持自定义词法高亮（词法着色）
  • 把“代码注释”、“关键字”、“字符串”、“数字”等语法要素用不同的颜色区分开
  • 根据“类名”、“函数名”、“变量名”等进行着色

快捷键

  • 支持快捷键（大多数都能做到）
  • 支持自定义快捷键

其他功能

  • 代码自动缩进、自动补全、动态提示
  • 支持跨平台

源代码管理工具（版本管理软件）

  • 简称“RCS”（Revision Control System）
  • 通过 RCS 的使用状况，可以看出一个开发团队在软件工程方面的成熟度

RCS有如下几种误用的情况：

  • 不正确的提交频度
    1. 很少进行代码的提交甚至没有提交
    2. 频繁提交代码
      每改完一个文件就提交一次，甚至把修改一半，尚【不能】编译通过的代码也提交到 RCS

正确的提交频度应该分两种情况：

  1. 在编写功能代码时，每完成一个功能点，经过自测后提交一次
  2. 在调试的时候，每修复一个 bug提交一次

这样能够保证提交到 RCS 的代码都是【能编译通过】的，并且业务逻辑上也是相对完整的

  • 提交时不写注释
  • 不做代码基线（Baseline）、不做代码分支（Branch）

每当一个版本发布（Release）并交付给用户使用时，都需要在 RCS 中制作一个基线，以便于进行相应的 bug 跟踪和补丁制作
诸如【做基线】之类的事情，属于团队的集体行为，需要由 Team Leader 牵头

用于调试/测试的运行辅助工具

开发机器上要有尽可能仿真（和用户的环境相似）的运行环境。
运行辅助工具能够有效地发现运行时的一些不正常的信息，让 bug 在交付给用户之前【尽早暴露】出来：

  • Web 开发人员要安装好常用的浏览器（至少包括 IE、Firefox、Chrome）。对于 IE，还得设置成“显示脚本错误通知”
  • 开发 Windows 应用程序要备好诸如：Depends（Visual C++自带）、Process Explorer等工具，便于查看进程、线程、动态库、堆内存等运行时信息
  • 手机应用开发至少得有目标系统的模拟器软件，真机更好
  • 进行跨平台方面的开发，诸如“VMware / VirtualBox”的虚拟化软件必不可少

善用自动化

什么是自动化

自动化是利用各种方式（主要是计算机）来帮你【自动完成】某些（枯燥、费时、无价值的）重复劳动

自动化的重要性

  • Perl 语言的创始人Larry Wall 曾经评价过程序员的三大美德，分别是：【懒惰、急躁、傲慢】
  • 懒惰，就是干尽量少的活，但是依然保质保量地完成工作
  • 如何才能偷懒？一个有效的办法就是【自动化】

怎么实现自动化

  • 观察你平时做的事情中，有哪些属于【重复】劳动
  • 评估这些重复劳动是否可以用某些工具来替代
  • 动手把这个工具实现出来

实现自动化的例子

Blog 订阅

如果经常访问博客，可以使用 Feed 订阅工具。在订阅工具中初始化URL（只要做一次），订阅工具会在有新帖子时通知你

调试程序

当程序行为不正常时，经常需要设置断点，然后单步跟踪代码，以便找出程序出错的源头。这个过程也有大量的重复劳动
可以通过程序断言（assert）来简化上述过程：

  • 每个 assert 好比一个【自动的】代码检查点，【每次】程序运行到 assert 处，如果你设置的逻辑条件不成立，它会立即终止程序并打印出相关信息（比如函数调用栈、文件行号等）
  • 在写代码的时候经常在一些【关键点】设置一些【条件恰当】的 assert，可以大量节约调试时间
    1. 在自测的时候，有70%-80%的逻辑错误会被 assert 暴露出来，所以改起来非常快
    2. 测试人员提交的Bug也有一半以上可以通过 assert 快速定位出错误的源头

自动化测试

测试人员重复劳动特别严重：

  • 不断重复做软件功能的验证操作
  • 发现bug后通知程序员改
  • 程序员改完，再次进行验证操作

尽量多使用自动化测试工具（比如 QTP）和测试脚本完成上述的软件功能验证操作。好处如下：

  • 节约时间
  • 提高效率
  • 在编写测试脚本的过程中学习新技术，提高个人能力
    例如：经常编写Python脚本进行网络和数据库测试，熟练之后尝试Python 开发

人肉自动化

出差订机票属于重复劳动：

  • 网上查航班
  • 看哪个航班折扣优惠
  • 选好航班后付钱
  • 去机场打印行程单
  • 出差回来填写报销单（报销单得找N个人签字）
  • 拿报销单与行程单找秘书报销

为了提高效率，把上述琐事交给秘书。出差只需要发邮件告诉秘书：要订某天某时的飞机到某地。经过这样的改革，部门里的人（除了秘书）都皆大欢喜
